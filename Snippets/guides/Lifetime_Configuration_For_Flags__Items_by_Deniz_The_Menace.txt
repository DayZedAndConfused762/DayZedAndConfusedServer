These explanations provide different angles to understand how the flag & lifetime settings interact to determine coverage within the Types.xml file based on the Global.xml file configurations.

2 ways of explaining this

1 - FlagRefreshFrequency Below Lifetime Setting
‚ñ∏ To ensure that items within the Types.xml file are covered according to the lifetime values set in the Global.xml file, the FlagRefreshFrequency must be set below the desired lifetime.
‚ñ∏ Essentially, the FlagRefreshFrequency determines how often the flags are refreshed, & for items to be covered within their specified lifetime, the flag must be refreshed more frequently than the lifetime duration.
‚ñ∏ This ensures that the flags accurately reflect the current state of the items within the Types.xml file.


2 - Lifetime Setting Above FlagRefreshFrequency
‚ñ∏ Another perspective is that the lifetime values set in the Types.xml file need to be above the FlagRefreshFrequency value specified in the Global.xml file.
‚ñ∏ The lifetimes define how long items are considered valid or covered, while the FlagRefreshFrequency determines how often the flags are updated to reflect these lifetimes.
‚ñ∏ Therefore, for the items to be appropriately covered within their defined lifetimes, the lifetimes must be longer than the frequency at which the flags are refreshed.
‚ñ∏ This ensures that the flags accurately represent the current state of the items over time.


Below is a simple guideline to help you grasp the various lifetimes in Vanilla types.xml & global.xml files.
This explanation highlights the default values covered & those not covered, whether a flag is raised or not.

The FlagRefreshMaxDuration of 3456000 seconds (40 days) will ensure that the flag will continue to refresh item/s for a maximum of 40 days

!========================================================================
*                   ‚ú® Example 1
*               üìç 1 WEEK SETUP (7 Days)

üìç IMPORTANT INFORMATION

To ensure that tents, fences, watchtowers, & any other items with lifetimes of 14 or 45 days are removed within a week if they don't have a flag, you'll need to adjust the lifetimes in the types.xml file.
This is particularly useful for removing abandoned player-built structures within a week's timeframe.


HOW TO DO THIS

Types.xml
‚ñ∏ Change all the lifetimes that are 1209600 (14 Days) to 604800 (07 Days)
‚ñ∏ Change all the lifetimes that are 3888000 (45 Days) to 604800 (07 Days)

üìç IMPORTANT INFORMATION

To ensure that tents, fences, watchtowers, & other items with lifetimes of 7, 14, & 45 days are removed within 3 days if they don't have a flag, you'll need to adjust the lifetimes in the types.xml & global.xml file.
This is particularly useful for removing abandoned player-built structures within 3 days of timeframe.


HOW TO DO THIS

Types.xml
‚ñ∏ Change all the lifetimes that are 604800 (07 Days) to 259200 (03 Days)
‚ñ∏ Change all the lifetimes that are 1209600 (14 Days) to 259200 (03 Days)
‚ñ∏ Change all the lifetimes that are 3888000 (45 Days) to 259200 (03 Days)

üìç IMPORTANT INFORMATION

Now, let's say you want to make sure people raise this flag often, at least once a week.
You can adjust how long the flag can stay up before it needs to be raised again.

‚ñ∏ Normally, the flag can stay up for a long time, like 40 days, before someone needs to raise it again.
‚ñ∏ If you want people to raise it more frequently, you can change this rule.
‚ñ∏ You can adjust a special setting called "FlagRefreshMaxDuration".
‚ñ∏ Instead of 40 days, you can set it to any time you want, the time is measured in seconds, so for this example we will use 7 days.


HOW TO DO THIS

Types.xml
‚ñ∏ Change all the lifetimes that are 1209600 (14 Days) to 604800 (07 Days)
‚ñ∏ Change all the lifetimes that are 3888000 (45 Days) to 604800 (07 Days)

Global.xml
‚ñ∏ Change the value to 7 Days <var name="FlagRefreshMaxDuration" type="0" value="604800"/> = 07 Days
